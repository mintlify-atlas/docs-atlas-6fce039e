The JKPSC Answer Evaluator uses Google's Gemini API to provide intelligent, rubric-based evaluation of civil services exam answers. This page explains the integration architecture, prompt engineering, and model selection strategy.

## Core Evaluation Flow

The evaluation logic is centralized in `src/lib/gemini.ts:495`, which orchestrates the following process:

1. **Prompt Building** - Subject-specific prompts are constructed using the `buildPrompt` function
2. **Worker Delegation** - Evaluation runs in a Web Worker to keep the UI responsive
3. **API Call** - Request is sent to Gemini's REST API with the prompt
4. **Response Parsing** - JSON response is extracted and validated with robust error handling

```typescript
// src/lib/gemini.ts:495
export const evaluateAnswer = async (params: EvaluationParams): Promise<EvaluationResult> => {
  const prompt = buildPrompt(params);

  // Try Web Worker first (keeps UI responsive)
  if (typeof Worker !== 'undefined') {
    try {
      const result = await runInWorker(params.apiKey, prompt, params.marks);
      return result;
    } catch (workerErr) {
      console.warn('[gemini] Worker failed, falling back to main thread:', workerErr);
    }
  }

  // Main-thread fallback
  return evaluateOnMainThread(params.apiKey, prompt, params.marks, params);
};
```

## Prompt Engineering

The `buildPrompt` function (`src/lib/gemini.ts:20`) routes to subject-specific prompt builders:

```typescript
export const buildPrompt = (params: EvaluationParams): string => {
  switch (params.subject) {
    case 'Essay': return buildEssayPrompt(params);
    case 'GS1': return buildGS1Prompt(params);
    case 'GS2': return buildGS2Prompt(params);
    case 'GS3': return buildGS3Prompt(params);
    case 'GS4': return buildGS4Prompt(params);
    default: return buildGS1Prompt(params);
  }
};
```

Each prompt builder follows a structured template:

### Prompt Structure

1. **Role & Context** - Sets the evaluator persona (e.g., "UPSC GS-1 Lead Examiner")
2. **Question & Answer** - Injects the candidate's submission
3. **Golden Rules** - Enforces strict ceiling (70% max for top answers) and generic penalties
4. **Evaluation Criteria** - Word limits, mark distribution, and syllabus headers
5. **Non-Negotiable Checklist** - Must-have elements (data, maps, articles, cases)
6. **PESTEL+ Dimensions** - Multidimensional analysis framework
7. **Scoring Matrix** - Detailed rubric for mark allocation
8. **JSON Output Schema** - Enforces structured response format

### JSON Output Instructions

The `jsonOutputInstructions` function (`src/lib/gemini.ts:31`) constructs a strict JSON schema that Gemini must follow:

```typescript
const jsonOutputInstructions = (params: EvaluationParams, categoryNames: { name: string; maxScore: number }[]) => {
  const ceiling = Math.floor(params.marks * 0.7);
  const categoriesJson = categoryNames
    .map(c => `    {"name": "${c.name}", "score": <max ${Math.floor(c.maxScore * 0.7)}>, "maxScore": ${c.maxScore}, "feedback": "<3 sentences: what was done well, what was missing (name specific terms/articles/cases), and what a topper answer would include.>"}`,)
    .join(',\n');

  return `## RETURN THIS EXACT JSON STRUCTURE (aim for ~2000 tokens of output):
{
  "overallScore": <number between 0 and ${ceiling}, NEVER higher>,
  "maxScore": ${params.marks},
  "categories": [
${categoriesJson}
  ],
  "keyPointsAnalysis": {
    "covered": ["<specific term/article/case mentioned by candidate>", ...],
    "missing": ["<critical term/article/case NOT mentioned and why it matters>", ...]
  },
  "improvementTips": [...],
  "modelAnswerStructure": "<Write a model answer blueprint of 150-200 words...>",
  "strengths": [...],
  "summary": "<2-3 sentence evaluation>"
}

INSTRUCTIONS:
- Be clinical and strict. Apply the 70% ceiling.
- Each category feedback: 3 sentences referencing specific parts of the answer.
- keyPointsAnalysis: 4-6 items each in covered and missing.
- improvementTips: 5 tips, each 1-2 sentences with specific actionable advice.
- Return ONLY valid JSON, no markdown or extra text.`;
};
```

This ensures the model returns structured, parseable feedback with category-wise scoring, key points analysis, and actionable improvement tips.

## Model Selection

The app uses **two different Gemini models** for different tasks:

### 1. `gemini-3-flash-preview` (Answer Evaluation)

**Used in:** `src/lib/gemini.ts:544` (main thread evaluation)

```typescript
const response = await fetch(
  `https://generativelanguage.googleapis.com/v1beta/models/gemini-3-flash-preview:generateContent?key=${apiKey}`,
  {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      contents: [{ parts: [{ text: prompt }] }],
      generationConfig: {
        temperature: 0.3,  // Low temperature for consistent scoring
        maxOutputTokens: 65536,  // Large output for detailed feedback
      },
    }),
  }
);
```

**Why this model?**
- Higher reasoning capability for complex rubric evaluation
- Better understanding of nuanced Indian civil services exam requirements
- Supports large output tokens (up to 65536) for comprehensive feedback
- Temperature set to 0.3 for consistent, predictable scoring

### 2. `gemini-2.5-flash` (OCR & Text Extraction)

**Used in:** `src/components/PdfUpload.tsx:195` (vision API for handwritten answer extraction)

```typescript
const response = await fetch(
  `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent?key=${apiKey}`,
  {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      contents: [{ parts }],  // parts = image data + OCR instructions
      generationConfig: {
        temperature: 0.1,  // Very low for accurate transcription
        maxOutputTokens: 16384,
      },
    }),
  }
);
```

**Why this model?**
- Excellent multimodal capabilities (vision + text)
- Faster inference for OCR tasks
- Lower temperature (0.1) for faithful transcription
- Smaller output tokens (16384) sufficient for question/answer extraction

## Error Handling

The app implements robust error handling for API failures:

```typescript
if (!response.ok) {
  const error = await response.json();
  if (response.status === 400 && error.error?.message?.includes('API key')) {
    throw new Error('Invalid API key. Please check your Gemini API key.');
  }
  if (response.status === 429) {
    throw new Error('Rate limit exceeded. Please wait a moment and try again.');
  }
  throw new Error(error.error?.message || 'Failed to evaluate answer');
}
```

### Response Parsing with Fallbacks

The response parser (`src/lib/gemini.ts:586`) includes multiple fallback strategies:

1. **Standard JSON Parse** - Tries direct JSON parsing after stripping markdown fences
2. **Truncated Response Repair** - Detects incomplete JSON and attempts to complete it
3. **Bracket Balancing** - Auto-closes unclosed arrays and objects
4. **Regex Extraction** - Extracts fields individually if full parse fails

This ensures the app can still extract usable evaluation data even if Gemini's response is truncated or malformed.

## API Key Management

The API key is **never sent to any server** except Google's Gemini API. The app:

- Stores the key locally in `localStorage` (plain or encrypted)
- Optionally encrypts the key with AES-GCM using a user PIN (see [Storage](/technical/storage))
- Validates the key on first use and shows clear error messages for invalid keys

## Performance Optimization

The app uses **Web Workers** to offload evaluation to a background thread:

```typescript
function runInWorker(apiKey: string, prompt: string, marks: number): Promise<EvaluationResult> {
  return new Promise((resolve, reject) => {
    const worker = new Worker(
      new URL('../workers/evaluationWorker.ts', import.meta.url),
      { type: 'module' },
    );

    worker.onmessage = (e) => {
      worker.terminate();
      if (e.data.type === 'result') {
        resolve(e.data.result as EvaluationResult);
      } else {
        reject(new Error(e.data.message ?? 'Worker evaluation failed'));
      }
    };

    worker.postMessage({ apiKey, prompt, marks });
  });
}
```

This keeps the UI responsive during the 5–15 second evaluation process.
